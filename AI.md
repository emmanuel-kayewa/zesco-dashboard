# AI / LLM in ZESCO Dashboard — How It Works

This document explains how the AI features in this application work end-to-end (frontend UI → Laravel routes/controllers → queue jobs → AI provider layer → caching), based on the current codebase.

## 1) What “AI” means in this app

The app implements a **provider-agnostic AI layer** with two supported backends:

- **Ollama (local inference)** — default. The app talks to an Ollama server over HTTP.
- **OpenAI (cloud)** — optional. The app calls the OpenAI Chat Completions API.

The AI is used for:

- Executive “AI Insights” summary and an “Ask AI” chat on the global AI page.
- A PP (Planning & Projects) **scoped assistant** (drawer) that only answers using PP portfolio data.
- KPI import **auto-categorization/enrichment** (admin-only).
- A scheduled **weekly digest** (email + optional WhatsApp) generated by AI.

## 2) Main user entry points (UI)

### 2.1 AI Insights page (global)

- URL: `/ai`
- Backend: `AiInsightsController@index` renders `resources/js/Pages/AI/Insights.vue` via Inertia.
- What it exposes:
  - **Ask AI** → POST `/api/ai/query`
  - **Executive AI Summary** → POST `/api/ai/executive-insights` (optionally `{ fresh: true }` to bypass cache)
  - **Async support**: if the server is using a non-`sync` queue driver, requests may return a `task_id` and the page polls until completion.

Implementation:
- UI: `resources/js/Pages/AI/Insights.vue`
- Polling logic: implemented inline in that page (and also available in the shared composable, see below).

### 2.2 PP AI Assistant drawer (scoped)

- Component: `resources/js/Components/AI/AiDrawer.vue`
- It sends scoped questions to **PP-only AI endpoint**:
  - POST `/api/ai/pp/query` with:
    - `question: string`
    - `scope: object` (required; determines the PP context)
    - `history: [{ role: 'user'|'assistant', content: string }]` (last 12 messages)

The drawer also checks provider status:
- GET `/ai/status` (shows provider identifier if available)

The shared async helper is centralized in:
- `resources/js/Composables/useAiTasks.js` (`aiPost()` + `pollTask()`)

### 2.3 Admin KPI Import AI enrichment

Admin route group exposes:
- POST `/admin/kpi-import/ai-enrich`

This endpoint calls AI to suggest category/unit/trend/targets for uploaded KPI rows.

Implementation:
- Controller: `app/Http/Controllers/KpiImportController.php` (`aiEnrich()`)
- AI logic: `app/Services/AiAnalysisService.php` (`categorizeKpis()`)

### 2.4 Weekly Digest (scheduled / operational)

- Artisan command: `dashboard:weekly-digest`
- Scheduled in `routes/console.php`:
  - weekly on Monday at 07:00
- Generates an AI digest and sends:
  - Email to roles: `admin`, `executive`, `directorate_head`
  - Optional WhatsApp template messages (if enabled/configured)

Implementation:
- Command: `app/Console/Commands/WeeklyDigestCommand.php`
- AI generation: `AiAnalysisService::generateWeeklyDigest()`
- Email template: `app/Mail/WeeklyDigestMail.php` (view: `emails.weekly-digest`)

## 3) HTTP routes and controllers (backend)

All AI routes are defined in `routes/web.php` inside the protected routes group, meaning:

- In non-local environments: `auth` + `session.timeout` middleware is applied.
- In local environments: a `dev.autologin` middleware is used instead of `auth`.

### 3.1 Page routes

- GET `/ai` → `AiInsightsController@index`
- GET `/ai/status` → `AiInsightsController@status`

### 3.2 API routes

Under prefix `/api/ai`:

- POST `/api/ai/executive-insights` → `AiInsightsController@executiveInsights`
- POST `/api/ai/query` → `AiInsightsController@query`
- POST `/api/ai/pp/query` → `AiInsightsController@ppQuery`
- POST `/api/ai/explain-anomaly` → `AiInsightsController@explainAnomaly`
- POST `/api/ai/recommendations` → `AiInsightsController@recommendations`
- POST `/api/ai/predict-breach` → `AiInsightsController@predictBreach`
- POST `/api/ai/clear-cache` → `AiInsightsController@clearCache`
- GET  `/api/ai/task/{taskId}` → `AiInsightsController@pollTask`

> Note: not every endpoint is currently wired to a visible UI control, but they are available and implemented.

### 3.3 PP access control

The PP-scoped endpoint uses explicit access checks:

- `AiInsightsController::ppQuery()` calls `enforcePpAccess()`.
- Rules:
  - Admins always allowed.
  - Otherwise user must be `directorate_head` AND belong to directorate code `PP`.

This is in addition to the route-level authentication.

## 4) Sync vs async: the job + polling architecture

AI calls can take minutes (especially large local models). The app supports both:

- **Synchronous** execution (when queue driver is `sync`)
- **Asynchronous** execution (when queue driver is anything else)

### 4.1 Decision logic

In `AiInsightsController`:

- `shouldAsync()` returns `config('queue.default') !== 'sync'`.
- Each endpoint:
  - Validates the request
  - Checks AI availability
  - Either dispatches an async task or performs the AI call directly.

### 4.2 Async request lifecycle

1. Controller generates a `taskId` (UUID).
2. It seeds cache key `ai_task:{taskId}` with status `queued` (TTL: 30 minutes).
3. It dispatches `ProcessAiRequest` job with:
   - `taskId`
   - `method` (string enum like `executiveInsights`, `ppQuery`, …)
   - `params` (payload for that method)
4. Frontend receives `{ async: true, task_id }` and begins polling:
   - GET `/api/ai/task/{taskId}`
5. The job updates cache status:
   - `running` → `completed` (with `result`) or `failed`.

### 4.3 Job implementation details

- Job class: `app/Jobs/ProcessAiRequest.php`
- Timeout: `900` seconds (15 min)
- Retries: `3`
- Backoff: `30s, 60s, 120s`
- Result storage: cache key `ai_task:{taskId}` (TTL: 30 minutes)

This design prevents PHP web requests from being killed by `max_execution_time` while a model is generating.

### 4.4 Frontend polling helpers

- Global AI page includes its own `pollTask()` / `aiPost()`.
- The PP drawer uses the shared composable:
  - `resources/js/Composables/useAiTasks.js`

Both behave the same:

- POST request
- If response contains `{ async: true, task_id }` then:
  - poll every 3 seconds
  - timeout after 900 seconds

## 5) What data is sent to the model (context building)

All “real intelligence” comes from **structured context** gathered from the application database/services and embedded into the AI prompt.

The core service is:

- `app/Services/AiAnalysisService.php`

It:

- Pulls relevant data from models and existing dashboard services
- Builds a system prompt + a user prompt containing JSON data
- Calls `AiProviderManager::provider()->chatWithJson(...)`

### 5.1 Executive Insights context

Method:
- `AiAnalysisService::generateExecutiveInsights(bool $fresh)`

Data gathered includes:

- Executive summary from `DashboardService::getExecutiveSummary()`
- All active directorates and their active KPIs
- Latest KPI entry per KPI+directorate (value, change %, status, deadlines)
- Up to 20 unread alerts (type/severity/title/message)
- Wayleave/survey processing breakdown per directorate

Caching:
- Key: `ai_executive_insights`
- TTL: `dashboard.ai.cache.executive_insights` (default 1440 minutes)
- Adds metadata to the stored result: `generated_at`, `provider`

### 5.2 Natural language query (global “Ask AI”)

Method:
- `AiAnalysisService::answerQuery(string $question)`

Context:

- Executive summary
- List of directorates
- List of available KPIs
- Unread alerts (latest 10)
- Wayleave/survey processing data

The system prompt instructs:
- Use provided data only
- Be specific with numbers and names
- Return valid JSON with answer + data points + confidence

### 5.3 PP-scoped query (PP drawer)

Method:
- `AiAnalysisService::answerPpScopedQuery(array $scope, string $question, array $history = [])`

Key properties:

- Strictly scoped: “ONLY using the provided PP portfolio data”
- Explicitly disallows cross-directorate comparisons from the PP view
- Adds limited conversation history text (up to 10 recent messages after sanitizing and truncating)

Scope payload

The controller requires:

- `scope.type` (string)
- optional `scope.pp_project_id`
- optional `scope.filters` (array)

Context builder:
- `AiAnalysisService::gatherPpQueryContext(array $scope)`

It uses `PpDashboardService` to fetch PP dashboard data depending on `scope.type`:

- `pp_project`: pulls a single project’s detail and slices lists (milestones/financials/risks/…)
- `pp_grid_studies`: grid studies KPIs + breakdown samples
- `pp_explorer`: explorer projects + breakdown samples
- default: PP portfolio overview

To avoid huge prompts, it:

- caches context via `Cache::remember()` using a SHA1 key over scope parameters
- slices arrays and also checks the final JSON length (reduces further if > 60k)

TTL:
- `config('dashboard.ai.cache.pp_context', 10)` minutes (defaults to 10 unless explicitly added to config)

### 5.4 Explain KPI anomaly

Method:
- `AiAnalysisService::explainAnomaly(int $kpiId, int $directorateId)`

Context includes:

- Last 12 KPI entries for the KPI+directorate
- Latest values for other KPIs in the same directorate
- Recent incidents (last 3 months, if model exists)
- Top risks by score

Caching:
- `ai_anomaly_{kpiId}_{directorateId}`
- TTL: `dashboard.ai.cache.anomaly_explanation` (default 720 minutes)

### 5.5 Directorate recommendations

Method:
- `AiAnalysisService::suggestActions(int $directorateId)`

Context includes:

- Directorate detail from `DashboardService::getDirectorateDetail()`
- KPI deadline information (days remaining, overdue flags)

Caching:
- `ai_recommendations_{directorateId}`
- TTL: `dashboard.ai.cache.recommendations` (default 720 minutes)

### 5.6 KPI deadline breach prediction

Method:
- `AiAnalysisService::predictDeadlineBreach(int $kpiId, int $directorateId)`

Context includes:

- KPI target/deadline metadata
- All historical KPI entries (value + date)
- Linear forecast from `DashboardService::forecastKpi()`

### 5.7 KPI import categorization (admin)

Method:
- `AiAnalysisService::categorizeKpis(array $rawKpis)`

Inputs:
- Array of KPI name/description/category/unit (raw from uploaded file)

Outputs:
- Suggested cleaned KPI name, category, unit, trend direction, targets, thresholds, currency code, etc.

The categories list is taken from:
- `config('dashboard.kpi_categories')`

### 5.8 Weekly digest

Method:
- `AiAnalysisService::generateWeeklyDigest()`

Context includes:

- KPI movements over the last week (entries since 7 days ago grouped by directorate)
- Alerts created in the last week

## 6) AI provider layer (Ollama vs OpenAI)

All model calls go through:

- `app/Services/AI/AiProviderManager.php`
- Interface: `app/Services/AI/AiProviderInterface.php`

Provider is selected by config:

- `config('dashboard.ai.provider')` (from env `AI_PROVIDER`)

### 6.1 Ollama provider (local)

Implementation:
- `app/Services/AI/OllamaProvider.php`

Key behavior:

- Uses streaming (`/api/chat` with `stream: true`) to avoid long “no output” timeouts.
- Attempts to keep the configured model loaded and unload competing models:
  - GET `/api/ps` to see loaded models
  - POST `/api/generate` with `keep_alive: 0` to unload
  - POST `/api/generate` with `keep_alive: '10m'` to preload
- Retries connection issues and waits for recovery (`/api/tags` polling).

Config:

- `OLLAMA_URL` (default `http://localhost:11434`)
- `OLLAMA_MODEL` (default `qwen2.5:72b`)
- `OLLAMA_TIMEOUT` (default 600 seconds; used as per-chunk read timeout in streaming)

### 6.2 OpenAI provider (cloud)

Implementation:
- `app/Services/AI/OpenAiProvider.php`

Key behavior:

- Uses `/chat/completions`
- For JSON responses:
  - adds an instruction “respond with valid JSON only”
  - sets `response_format: { type: 'json_object' }`

Config:

- `OPENAI_API_KEY`
- `OPENAI_MODEL` (default `gpt-4`)
- `OPENAI_BASE_URL` (default `https://api.openai.com/v1`)
- `OPENAI_TIMEOUT` (default 60 seconds)

Operational note:
- When using OpenAI, the dashboard context payload is sent to a third-party API.

## 7) Caching strategy

There are two major cache usages:

1) **Async task results**
- Keys: `ai_task:{uuid}`
- TTL: 30 minutes (controller + job)

2) **AI analysis caches**
- Uses cache tags (`tags(['ai'])`) if the cache driver supports tagging.
- Falls back to plain cache if tagging is not supported.

Important cache keys:
- `ai_executive_insights`
- `ai_anomaly_{kpiId}_{directorateId}`
- `ai_recommendations_{directorateId}`
- `ai_pp_context:{sha1(scope)}` (via `Cache::remember`)

Clearing:
- POST `/api/ai/clear-cache` calls `AiAnalysisService::clearCache()`.

## 8) Availability / health checks

### 8.1 Backend availability logic

- `AiAnalysisService::isAvailable()` checks:
  - `dashboard.ai.enabled` is true
  - the configured provider `AiProviderManager::isAvailable()` returns true

### 8.2 Status endpoint

- GET `/ai/status` returns:
  - `available` (boolean)
  - `provider` (identifier like `openai:gpt-4`)
  - `configured_provider`
  - `enabled`
  - `async` (whether queue driver is not `sync`)

The PP drawer uses this to display provider info.

## 9) Validation and guardrails (security + correctness)

### 9.1 Request validation

The controller validates input for each endpoint. Examples:

- `question` max length: 1000 chars
- `history` max messages: 12; each message max 1000 chars
- IDs must exist in DB (`exists:kpis,id`, `exists:directorates,id`, …)

### 9.2 Authorization

- AI routes are inside the protected group in `routes/web.php` (auth/session timeout in non-local).
- Admin KPI import AI enrichment is under the admin-only route group.
- PP AI endpoint has additional PP-specific access enforcement.

### 9.3 “Answer only from data” constraints

The prompts explicitly instruct the model to:

- Use only the provided JSON data
- Not invent figures
- Return JSON only

This is implemented by:

- Carefully building the user prompt with “AVAILABLE DATA” / “PP DASHBOARD DATA”
- For JSON mode, adding strict “JSON only” instructions
- Using provider-level JSON response mechanisms (Ollama `format: json`, OpenAI `response_format`)

## 10) Operations: what you must run for AI to work

If you want AI features to work reliably, ensure:

- Queue worker is running (unless using `QUEUE_CONNECTION=sync`):
  - `php artisan queue:work --tries=3`
- Ollama server is running (default provider):
  - `ollama serve`
  - `ollama pull qwen2.5:72b`
  - `ollama pull qwen2.5:14b`

See `commands.md` for a consolidated “services to run” checklist.

Infrastructure note:
- The included infrastructure request docs indicate local inference may require a dedicated GPU server.

## 11) Extending AI (adding a new capability)

To add a new AI feature in a consistent way:

1. Add a method to `AiAnalysisService` that:
   - gathers context
   - defines a strict system prompt
   - calls `$this->ai->provider()->chatWithJson(...)`
2. Add a controller endpoint in `AiInsightsController` (validate + availability + async/sync wrapper).
3. Add a route under `/api/ai/...`.
4. Update the frontend to call the endpoint using `useAiTasks().aiPost()`.

To add a new provider:

1. Implement `AiProviderInterface`.
2. Register it in `AiProviderManager::resolveProvider()`.
3. Add configuration keys under `config/dashboard.php`.
